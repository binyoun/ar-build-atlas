Native and engine-based AR

Full game engines and platform SDKs, distributed through an app store rather than a URL. This is where the ceiling on visual fidelity and interaction complexity is highest, and also where vibecoding stops covering the whole job: a real editor (Unity, Unreal, Xcode) does work a text-only coding agent cannot fully replace.

Unity + AR Foundation Cross-platform game engine (C#) with a unified layer over ARKit and ARCore, so one project targets both iOS and Android. Free up to a revenue and funding threshold, then a paid plan Medium to high: the engine, its editor, and C# all need learning together App store, or a sideloaded build Yes, once installed Medium. An agent can write solid C# scripts, but scene assembly, prefab wiring, and inspector settings happen by hand in the Unity Editor. Not yet used
Unity + Vuforia An image and object-tracking plugin for Unity, useful when AR Foundation's built-in tracking is not precise enough for a specific physical object. Adds a separate license cost on top of Unity Medium, on top of Unity's own curve Same as Unity above Yes, once installed Medium, same constraint as Unity above. Not yet used
Niantic Lightship ARDK A native SDK from Niantic Spatial (with a Unity plugin) adding spatially aware AR beyond basic plane detection: semantic segmentation of the camera feed (sky, ground, buildings, people), meshing, and depth. Free tier, usage-based paid tiers Medium to high, an additional SDK layered on top of Unity or native iOS/Android App store Yes once installed, though some features depend on cloud services Medium, same constraint as Unity above: scripts are agent-friendly, engine wiring is not. Not yet used
Unreal Engine (AR) C++ and Blueprints, aimed at the highest-fidelity real-time visuals, with an AR framework layered over ARKit/ARCore. Free until a revenue threshold, then a royalty High. The heaviest engine here, with the steepest ramp. App store Yes, once installed Low. Blueprint visual scripting does not map well to a text-based coding agent, and C++ plus the editor is a heavier lift than the other options here. Not yet used
ARKit + RealityKit Apple's native AR framework and 3D renderer, written in Swift, for iOS, iPadOS, and visionOS. Free, but needs a Mac plus a paid Apple Developer account (about 99 USD a year) to distribute Medium. Swift itself is approachable; Xcode's project and interface tooling is not. App Store, or ad-hoc install Yes, once installed Medium. An agent can write Swift well, but testing needs a physical iPhone or simulator and a Mac, which is a real barrier outside that ecosystem. Not yet used
ARCore + Kotlin Google's native AR framework for Android, in Kotlin or Java, the Android counterpart to ARKit. Free, needs an Android Developer account (one-time fee) to publish Medium, similar profile to ARKit Google Play, or a sideloaded APK Yes, once installed Medium, same profile as ARKit above. Not yet used
OpenXR An open, cross-platform standard (not a single vendor's product) for VR/AR APIs, meant to let one codebase target multiple headsets and runtimes without separate vendor-specific code paths. Free, an open specification with open implementations High. Usually worked with through an engine (Unity, Unreal) rather than raw. Depends on the target platform's app store Yes, once installed Low to medium, same constraint as whichever engine it is paired with. Not yet used
Snapdragon Spaces Qualcomm's AR developer platform for Snapdragon-powered smart glasses and headsets: hand tracking, plane detection, and image tracking as a native SDK aimed at emerging XR hardware built on Qualcomm chips. Free SDK; hardware determines the real cost Medium to high App store or sideload, tied to Snapdragon-based devices Yes, once installed Medium, same constraint as Unity/native above. Not yet used
None of the projects behind this atlas use this category yet. Everything shipped so far has stayed inside the browser. If a future piece genuinely needs app-store distribution, offline installation with no network at all, spatial features WebGL cannot reach (mesh occlusion, semantic segmentation), or targets emerging headset hardware, this is the honest next step, and it is worth treating as a real scope change rather than something to vibecode the same way as a WebAR project.
