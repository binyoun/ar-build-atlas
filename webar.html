WebAR: augmented reality that runs at a URL

No app store, no install, no account. A visitor scans a QR code or taps a link and the camera, tracking, and 3D all run inside the phone browser. This is the category every shipped project below actually used, so the "used in" column is the most filled in on this page.

Option What it is Cost Learning curve Distribution Offline capable Vibecoding fit Used in
Three.js A JavaScript 3D library over WebGL. No AR-specific layer built in; you supply your own camera feed, tracking, and anchoring. Free, open source Steep at first (matrices, shaders, render loop), then very flexible URL, any modern browser Yes, if all assets are self-hosted rather than pulled from a CDN at runtime High. Plain text files, exactly what a coding agent is best at. Uncanny Garden, Touch of Water, gelora-soil (vanitas)
A-Frame + MindAR A-Frame is a declarative, HTML-like layer over Three.js. MindAR adds image and face tracking on top. Free, open source Gentle. Scenes read almost like markup rather than 3D code. URL Yes, if self-hosted High. Config-driven setups (one entry per work) fit an agent well. Elemental WebAR (RMIT teaching kit)
Babylon.js A JavaScript 3D engine in the same territory as Three.js, but with more built in out of the box: a hosted node material editor, physics, and animation curve tooling. Free, open source Medium to steep, similar to Three.js, with less that needs to be hand-built URL Yes, if self-hosted High. Same plain-JS profile as Three.js. Not yet used
PlayCanvas A WebGL game engine with both a hosted, collaborative visual editor for scene assembly and a code layer for behavior, sitting between Three.js's roll-your-own approach and a full desktop engine. Free tier, paid tiers for private projects and team seats Medium. The editor covers layout, code covers logic. URL, either self-hosted export or a PlayCanvas-hosted link Yes, if exported and self-hosted Medium. The code layer is agent-friendly, but scene setup lives in their hosted editor, not in files an agent can edit directly. Not yet used
model-viewer Google's <model-viewer> web component: one HTML tag drops a 3D model into a page and hands off to the phone's native AR viewer (Scene Viewer on Android, AR Quick Look on iOS) instead of running its own AR session. Free, open source Very low, one HTML tag URL for the page; the AR view itself opens the OS-level viewer app The model file can be self-hosted; the OS viewer itself is a system component you do not control Medium for the web page itself, but there is no custom AR interaction to build since the OS viewer owns that part entirely. Not yet used
WebXR Device API The browser's own native immersive-ar session API: real 6dof tracking and hit-testing, no third-party library required. Free Medium. The API itself is small, but browser support gaps force workarounds. URL, Chrome for Android only in practice; no iOS Safari support as of this writing Yes Medium. The gap is platform support, not code complexity. Not yet used
8th Wall A hosted SDK (now under Niantic Spatial) that adds robust 6dof world tracking and image tracking to the browser, including on iOS Safari where raw WebXR does not reach. Subscription, tiered by traffic and features Medium. Mostly wiring their JS APIs and camera pipeline rather than building tracking from scratch. URL No. Requires their runtime and a license check at load time. Medium. Still text-based code, but you are working inside their SDK's constraints rather than a fully open stack. Not yet used
A recurring lesson across the projects above: MediaPipe's wasm runtime and model files are CDN-hosted by default. Venue wifi that cannot reach jsdelivr or Google's storage will break hand or face tracking silently. Self-hosting those files under public/ is the fix, and gelora-soil already does this.
